Council Report — Skincare AI Orchestration

Question: Is a photo-based product scanner + AI-orchestrated daily skincare routine (replacing the manual AM/PM checklist) a sound design for a personal health tracker? Stack: React + Flask + SQLite on Fly.io, Claude Sonnet 4.6. Clinical goal: treat active breakouts + fade PIH.
5 advisors · 5 peer reviews · Chairman synthesis · 2026-05-24
The One Thing to Do First

Open fly.toml and backend/models.py simultaneously and confirm two things: (1) a persistent volume is mounted so the SQLite database survives redeployment, and (2) the workout log table has a time-of-day field. If either is missing, fix it before writing a single line of skincare feature code — the routine engine's primary input and its entire data persistence model both depend on what you find there.

⚖️

Chairman's Verdict

✅ Where the Council Agrees

The photo-to-ingredient pipeline is the load-bearing failure point. Every advisor flagged it: Claude vision will misclassify ingredients, and every downstream rule runs on that classification. A wrong read doesn't produce a suboptimal recommendation — it produces a confidently wrong one.

Hardcoded rules must not live inside LLM prompts. "Max 2 medicated washes/24h" is a deterministic constraint, not a reasoning task. Putting it in a prompt means it is re-interpreted on every call, subject to hallucination, and untestable. These belong in Python logic, with Claude handling explanation only.

Medical adjacency requires a persistent disclaimer. The routine engine issues guidance for active breakouts and PIH — that's clinical territory. A one-time modal is not sufficient. The disclaimer must live in the routine UI itself, every time the routine is displayed.

The feedback loop is absent and it matters. Without tracking whether recommendations are helping or hurting, the system is an open loop. For a clinical goal, the user has no signal if the app makes things worse.

⚡ Where the Council Clashes

LLM for routine generation — yes or no? The First Principles advisor says cut it entirely (50-line decision function). The Executor treats LLM involvement as reasonable with guardrails. The Expansionist wants more LLM, not less.

The correct read: both are partially right. The decision tree (wash order, product selection, timing rules) should be deterministic Python. The LLM's role is legitimate but narrower: product identification at ingest, natural-language explanation, interaction warnings. Route the LLM around safety-critical logic, not through it.

Outcome tracking — build now or later? The Expansionist argues for a closed-loop feedback system from day one. Peer reviews push back: building a feedback loop on unreliable ingredient data and missing checklist persistence means optimizing noise. Correct answer: fix the foundation first, then instrument the feedback loop. The destination is right; the prerequisites are wrong.

🔍 Blind Spots the Council Caught

🚨
Fly.io SQLite volumes are ephemeral by default. Without an explicit persistent volume mount in fly.toml, every redeployment wipes the product inventory. The routine engine then silently fails — not with an error, but by generating routines against an empty product table. Highest-severity infrastructure risk in the design.
🔄
Routine idempotency is unresolved. If a user checks off steps, closes the app, and reopens — does Claude regenerate a potentially different routine or restore the saved one? For a clinical compliance goal, regeneration-on-open silently corrupts adherence tracking. Routine must be generated once per day, persisted, and restored on reload.
💡
Claude vision must run at ingest time only, never at display time. Re-scanning stored image blobs on every page load multiplies costs and stalls UX. Scan once on upload, store structured results, render from structured data thereafter.
⏰
Workout log time-of-day field may not exist. The rule "post-cardio → BP wash first" requires knowing when today's workout happened relative to now. The log likely stores date, not time-of-day. If absent, the primary routine-personalization trigger cannot be evaluated.
📉
Token cost and latency scaling. Sending the full product inventory + workout log + all rules on every request means latency and cost grow with inventory size. A pre-filter step — selecting relevant products before the Claude call — is necessary architecture nobody proposed.
⚡
LLM availability is a daily UX dependency. This checklist runs every morning. If Claude is slow, rate-limited, or down, the user gets nothing. Pre-generate the routine on workout log save (not routine open), with a deterministic fallback for the failure case.

🎯 The Recommendation

Build this — the concept is sound and the clinical goal is achievable — but restructure the architecture before writing feature code. Three distinct layers:

Layer 1 — Ingest (LLM-assisted, runs once): User photographs product → Claude vision extracts brand, type, and active ingredients → structured data written to SQLite. Image blob stored, never re-scanned. Low-confidence scans surface a manual entry fallback — not a hidden confidence score.

Layer 2 — Routine generation (deterministic Python, runs once per day): A Python function reads today's workout log (type + time-of-day), queries structured product inventory, and runs the rule set as explicit conditional logic. Output is persisted to a daily_routines table with a date key. Claude is called once, after the decision tree runs, to produce a natural-language explanation. Not to generate the routine itself.

Layer 3 — Rendering (no LLM, idempotent): The routine UI reads from daily_routines for today's date. If a routine exists, render it and restore checklist state. If not, trigger generation. No LLM calls at render time.

Defer the outcome-tracking feedback loop until the foundation is stable and checklist completion data has been accumulating for at least two weeks. Add a persistent, non-dismissible disclaimer in the routine UI header: "Suggestions based on general dermatological patterns. Consult a dermatologist for your specific condition."

Full Advisor Responses
The Contrarian
▼
The fatal flaw is the liability/safety gap dressed up as a feature. You're building a medical recommendation system while calling it a "routine generator." Those hardcoded dermatological rules aren't just UX decisions — they're clinical guidance. "Post-cardio BP wash first" and "max 2 medicated washes/24h" are the kind of instructions a dermatologist gives, not an app. When Claude misidentifies an active ingredient (and it will — skincare label photography is notoriously ambiguous, with overlapping terminology like "salicylic acid" vs "sodium salicylate" vs "willow bark extract"), the system confidently recommends something that could actively worsen active breakouts or cause chemical burns in combination.

Three specific failure modes: (1) Ingredient misclassification — Claude vision will hallucinate or misread small-print INCI lists under poor lighting. Database gets corrupted with wrong flags, and the routine engine silently makes bad decisions downstream. (2) Interaction blindness — the rules handle washing sequence but completely ignore dangerous combinations (retinoids + vitamin C, benzoyl peroxide + AHAs). No ingredient-interaction layer. For someone treating active breakouts, this isn't theoretical risk. (3) No feedback loop — zero mechanism to learn recommendations are making things worse.

Add explicit "this is not medical advice" guardrails with forced acknowledgment. Letting Claude read a workout log and prescribe a medicated wash sequence for someone with active breakouts is one bad recommendation away from a real skin barrier failure.
The First Principles Thinker
▼
The framing conflates two distinct problems with different optimal solutions. Problem 1: Product knowledge capture. The user wants to know what's in their products. This is a solved problem — a lookup against an ingredient database (INCI decoder, CosDNA) is more accurate than Claude vision guessing from a photo. The photo-to-ingredient pipeline introduces hallucination risk on exactly the data that drives the safety rules downstream. A bad ingredient read corrupts everything after it.

Problem 2: Routine sequencing. The hardcoded dermatological rules are doing real work — but they're hardcoded in a prompt, not in code. That's the wrong layer. Rules like "max 2 medicated washes/24h" should be enforced in deterministic logic, not inferred by an LLM each time. The routine decision tree is ~5 variables (time of day, workout type, sweat level, active ingredients on hand, UV). This is a 50-line decision function, not an LLM problem. The LLM adds latency, cost, and non-determinism to something that should be fast and predictable.

Use deterministic logic for routine generation. Use Claude only for product identification (ingredient extraction) and user-facing explanation ("why this order today"). Store ingredients as structured data, not a blob. Then the clinical rules are testable, auditable, and don't drift.
The Expansionist
▼
The design is sound but undershooting the opportunity significantly. You're building a context-aware dermatological AI that reads biometrics, inventory, and activity data to generate personalized protocols. The hardcoded rules are a floor, not a ceiling — Claude should reason dynamically from those principles, not just pattern-match against a lookup table. The real upside is adaptation over time.

What's missing: (1) Outcome tracking loop — add a weekly skin check-in (1-5 breakout severity + PIH improvement rating) and now Claude can correlate routine compliance + product combos + workout patterns against outcomes. That's a closed-loop optimization system. (2) Ingredient conflict detection across the full inventory — use extracted actives for cross-product interaction warnings, not just individual classification. (3) Streak/compliance analytics feeding into existing habit and mood modules already in the app.

The architecture is right. The SQLite photo blob + JSON routine + product inventory is a solid data foundation. Build the feedback loop from day one — retrofitting outcome tracking after months of data is painful.
The Outsider
▼
The design assumes users photograph products correctly, but people are bad at this. Blurry labels, angled shots, partial ingredient lists — Claude vision will confidently misidentify things. A BP wash misread as a gentle cleanser could mean someone applies benzoyl peroxide twice in quick succession. No fallback UI is shown for low-confidence scans.

The hardcoded rules are presented as universal. They aren't. "Post-cardio → BP wash first" ignores that BP degrades with sweat and UV, that some users cycle BP to avoid resistance, and that people with sensitive skin can't tolerate it daily regardless of workout type. The rules will be wrong for a meaningful slice of users, and users won't know they're wrong because the AI presents them with authority.

The photo blob in SQLite is a red flag: Fly.io SQLite with binary blobs doesn't scale and adds significant DB size over time. Those photos may contain prescription medications visible in the background. There's no mention of encryption or deletion policy.

Biggest concern: the system conflates routine suggestion with medical-adjacent advice. Needs a visible, persistent disclaimer — not a one-time modal, something present in the routine UI on every view.
The Executor
▼
The design is buildable. Claude vision for product ID is solid — one endpoint, one day of work. Routine generation is one Flask route. Here's where you'll hit walls:

First wall — workout timing dependency. "Post-cardio → BP wash first" requires knowing when the workout happened relative to now. The workout log almost certainly stores date but maybe not time-of-day. Check backend/models.py before writing a line of routine logic.

Second wall — Claude vision will fail on partial labels, non-English packaging, store-brand products. You need a fallback: if confidence is low, prompt manual active ingredient entry. Without this, the product inventory becomes silently incomplete.

Third wall — "max 2 medicated washes/24h" requires knowing what was actually checked off on previous routine checklists, not just what was generated. If you're not persisting checklist completion state, this rule cannot be enforced.

First Monday morning action: Open backend/models.py. Confirm (1) workout log has time-of-day field, (2) there's a table for routine checklist completions. If either is missing, add the schema migrations first.
