Full Design — Nutrition AI Draft Buttons

How all the pieces connect. Review each section and confirm before we write the spec.

Backend — /api/food/chat

System prompt contract

Add an instruction to the existing system prompt: when the reply mentions specific food items with concrete calorie/macro numbers, append a ```drafts fenced block after the human-readable text. The AI should NOT include the block when the reply is general advice (no specific numbers).

// appended to system prompt:
When your reply includes specific calorie/macro data for one or more foods, append this block on a new line after your text reply — do NOT include it otherwise:
```drafts
[{"food_name":"...","calories":0,"protein_g":0,"carbs_g":0,"fat_g":0}]
```
Backend — parsing

Strip + return structured data

After the LLM responds, backend uses a regex to find the ```drafts block, strips it from the reply, parses the JSON, and returns both fields. If no block is found, drafts is omitted from the response.

// Response shape when food data present:
{"reply": "A chicken breast 200g has ~330 cal...", "drafts": [{"food_name": "Chicken breast 200g", "calories": 330, "protein_g": 62, "carbs_g": 0, "fat_g": 7}]}

// Response shape for advice-only reply:
{"reply": "Try spacing meals 3-4 hours apart for better energy..."}
Frontend — CalorieChatbot

Draft buttons on AI messages

Messages get an optional drafts array stored alongside content. When rendered, each draft item gets its own button below the bubble.

🤖
A chicken breast 200g has ~330 cal, 62g protein, 0g carbs, 7g fat. A medium egg has 70 cal, 6g protein, 0g carbs, 5g fat.

Clicking calls onDraftEntry(draft) prop — lifts the data up to CalorieModule.

Frontend — CalorieModule → CalorieDayPanel

Pre-fill the inline form

CalorieModule holds a pendingDraft state. When set, it passes it as prefillEntry to CalorieDayPanel. The panel watches that prop: when it changes, it opens the inline food form with values pre-filled and defaults the meal type based on time of day (breakfast before noon, lunch before 3pm, dinner otherwise). The user can change the meal type and edit any field before hitting Add.

Scope

What's NOT changing

No new endpoints. No new components. No changes to the food log save flow, templates, or photo scanner. The chatbot's existing message history, suggestions, and scroll behavior are untouched. Only the system prompt, the response parser, message rendering, and the CalorieModule/CalorieDayPanel wiring change.
