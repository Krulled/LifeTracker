LLM Council Report

Nutrition Label Scan Feature — Design Decision

Question: Which entry-point approach best serves the nutrition label scan feature — auto-detect (A), two dedicated buttons (B), or tab toggle inside modal (C)? How should the overall feature be structured for a solo developer maintaining a personal mobile-first life tracker?
5 advisors · 5 peer reviews · 1 chairman synthesis 2026-05-23

Chairman's Verdict

Build Option B — but extract shared components first

✓ Where the Council Agrees

Option B (two buttons) wins on first principles and execution speed. Every advisor except The Expansionist landed there, and even The Expansionist's argument was about future architecture rather than a disagreement on day-one UX. The reasoning converges on a single insight: the user already knows what they're holding before they open the app. The classification decision costs zero cognitive effort when made by the human who has physical context, and it costs latency, API calls, and failure modes when delegated to AI.

All five advisors also agreed the edit/confirm step before saving is non-negotiable. OCR on nutrition labels is unreliable enough that saving without review is a health-data integrity risk.

⚡ Where the Council Clashes

The only real split is on Option A's potential. The Expansionist argues auto-classification is a trainable signal pipeline enabling future convergence (receipts, menus, barcodes). The other four treat this as premature architecture that introduces real failure modes on day one for speculative gains on day N.

The Expansionist is not wrong about the ceiling — the convergence vision is genuine. The problem is that it prices in the ceiling without pricing in the floor. A solo developer does not have the maintenance budget to build a confidence display, fallback flow, correction loop, and misclassification recovery path. That's not a small tax.

◉ Blind Spots the Council Caught

  • The edit/confirm step already partially absorbs Option A's failure cost — three reviewers independently caught this. Misclassification doesn't silently corrupt data; it means landing in the wrong modal and backing out. Annoying, not catastrophic. But on mobile mid-meal-prep, still unacceptable.
  • Component reuse is the real implementation decision, not button count. Does the label modal share the editable table + save function with the existing food modal? If yes, Option B's maintenance cost drops dramatically. If no, you've doubled your surface area.
  • Product cache for repeat scans. If a user scans a Clif Bar today, they shouldn't scan it again tomorrow. Storing parsed label results by product name and surfacing them as quick-tap options eliminates repeat friction. This is 20-30 lines against the existing SQLite schema.

→ The Recommendation

Build Option B. Two buttons: "Scan Meal" and "Scan Label" on the toolbar. Dedicated modal per type. Implementation priorities:

  • 1. Maximize component reuse. Label modal shares the edit/confirm table and save function with the food modal. Only the upstream parsing differs: food photo uses ingredient extraction + USDA lookup; label photo uses structured OCR. Both feed the same editable table → save flow.
  • 2. Invest in the label edit/confirm step, not the entry point. Nutrition label OCR is the real risk. Hallucinated serving sizes and swapped macro values kill trust. Pre-filled editable fields, one-tap focus on mobile.
  • 3. Add product cache on first ship. Save product name + macros after a successful scan. Surface recent products as quick-taps in the label modal header.

Do not build Option A now. If you want to hedge toward it, design upstream parsing as a swappable function so auto-routing can be layered in later without touching modal or save logic.

⚡ The One Thing to Do First

Before writing any new modal code, open the existing food photo modal and identify exactly which components can be extracted and reused: the editable table, the servings multiplier, the save handler. Extract those into shared components first. Then the label modal is a thin wrapper around shared infrastructure. That extraction is the actual work. The button is trivial.

Full Advisor Responses
The Contrarian Contrarian

Option A is the fatal flaw waiting to happen. Auto-classification will misfire constantly — bad lighting, partial labels, angled shots, a cereal box in the background of a meal photo. Every misclassification means the wrong modal launches, the user has to back out, retry, and now logging food is more annoying than just typing it manually. You've introduced latency and failure modes for zero UX gain.

The deeper problem with A: you're burning an extra AI call just to classify the photo type, before any actual extraction happens. Mobile users waiting on two sequential AI round-trips will abandon the feature.

Option C has a different fatal flaw: tabs inside a modal are a UI anti-pattern on mobile. Small tap targets, cognitive overhead, and users will constantly open the wrong tab and not notice until after they've taken the photo.

Option B is obviously correct. Two clearly labeled buttons eliminates every ambiguity. The real risk in this feature isn't which entry point you pick — it's label OCR accuracy. Nutrition labels have inconsistent formatting, small print, and the AI will hallucinate serving sizes or swap protein/carb values. Whatever you build, the confirmation/edit step before saving is non-negotiable.

The First Principles Thinker First Principles

The question being asked is "which UI pattern?" but the actual problem is minimum friction between "I have food in front of me" and "macro logged."

When someone is about to eat something from a package, they already know it's a package. The classification decision has already happened in their brain before they open the app. Auto-detection adds an AI inference step to eliminate a human tap — but the human already knows the answer. You're burning latency and compute to automate a decision the user already made for free.

The actual cost of "two buttons"? One tap. The cognitive overhead of choosing "Food" vs "Label" is near-zero because the user is holding the item. Option B is correct not because of UX theory — it's correct because it maps directly to reality: two genuinely different objects exist in the world, and two buttons represent that accurately without abstraction.

The deeper principle: don't automate decisions that have zero friction when made by a human who already has the context. Reserve AI for decisions where the human genuinely lacks information.

The Expansionist Expansionist

Option A is the sleeper here, and everyone is going to miss why. The real opportunity isn't just "which button layout is least annoying." It's that auto-classification is a trainable signal pipeline. Every correction the user makes is a labeled training example. A solo developer building this today is sitting on a dataset that could later power smart logging that requires zero deliberate mode selection ever.

The adjacent upside: once you have auto-classify working, the next step is receipt scanning, restaurant menu scanning, barcode lookup. None of those require new UI. They all collapse into the same "point camera, get nutrition data" interaction. Option B permanently forecloses that convergence by training users to think in modes.

Yes, auto-classification can misfire. But the fix is a one-tap correction, not a separate button. The confidence display ("Looks like a label — correct?") costs 20 lines of code and turns errors into engagement. Build A. The ceiling on a frictionless, camera-first logging experience is much higher than anyone on this council is pricing in.

The Outsider Outsider

Option B. The person standing in their kitchen holding a protein bar is in a specific mental state. They know what they're holding. They know it's a label. Why make them wait for AI to figure out what they already know, or make them tap into a modal and then decide between tabs? Every extra step between "I have food" and "it's logged" is a place the habit dies.

Option A sounds clever but adds a failure mode you'll spend time debugging. Option C hides the decision inside a modal — the user has to open something before they can even make the choice. That's worse than B.

Two buttons. Right there on the toolbar. The only real question is wording. "Scan Label" might not be immediately clear to someone new. Consider "Scan Package" or a barcode-adjacent icon. But that's a polish problem, not a structural one. B is faster to build, easier to maintain, and most honest about what the user is actually doing.

The Executor Executor

Option B. Ship it in an afternoon. Two buttons, two modals, done. The Monday morning steps are concrete: add a second toolbar button, copy the existing food photo modal, strip out the USDA lookup flow, replace it with label-parsing logic. You already have the modal scaffold, the Claude vision call pattern, and the food entry save function. You're wiring existing pieces together, not inventing architecture.

Option A sounds slick but adds a classification step that will fail on ambiguous photos. When it misclassifies, you need fallback UI anyway — so you've built Option B's complexity plus a failure mode. Option C is the worst of both worlds — one button, but you're managing tab state inside a modal, and the user still has to make a choice that's just been moved one click deeper.

The logging habit is the goal. Option B gives the user an explicit mental model: scan a plate, scan a label. No ambiguity, no accidental wrong branch. First step: add the "Scan Label" button to the toolbar JSX, wire it to an empty modal, confirm it renders on mobile.

Peer Review Highlights
Key themes from all 5 peer reviews

On the strongest response (split between B and D): Reviewers 1 and 2 named the First Principles Thinker as strongest for reframing the question — "the classification decision is already made in the user's head." Reviewers 3, 4, and 5 named the Contrarian for identifying OCR accuracy as the real risk surface and redirecting design attention to the edit/confirm step.

On the biggest blind spot (unanimous): All 5 reviewers flagged the Expansionist for treating a solo personal tracker as a scalable product. "Architecture astronautics," "category error," "pricing the upside at full value while pricing the downside near zero."

What all five advisors missed:

  • The existing edit/confirm step already absorbs most of Option A's misclassification cost (3 reviewers)
  • Component reuse vs. duplication is the actual implementation decision, not button count (3 reviewers)
  • A product-name cache to eliminate repeat scans for habitual items (1 reviewer)
  • Whether the toolbar is the right placement at all for a mobile-primary app (1 reviewer)
